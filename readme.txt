

TestBuilder — это система построения и выполнения тестовых сценариев в виде графа шагов.
Она построена таким образом, чтобы бизнес-логика тестов была полностью отделена от UI и внешней инфраструктуры.

Архитектура

Проект разделён на следующие слои:

Слой	Назначение
Domain	Бизнес-логика тестов: шаги, узлы графа, модели Modbus-устройств, мониторинг регистров
ViewModels	Представление графа для UI (редактор графа с Nodify)
Views	Avalonia UI, визуализация графа и элементов управления
Infrastructure	Работа с внешними системами: Modbus, COM-порты, серверные API, IO

Domain и Infrastructure изолированы от UI, что позволяет писать unit-тесты и интеграционные тесты без реального подключения к железу.

Domain
Основные сущности

ITestStep — интерфейс атомарного шага теста. Каждый шаг выполняет конкретную логику: запись, сравнение, проверку условий.

TestNode — узел графа. Управляет переходами между шагами, но не содержит бизнес-логики.

TestExecutor — интерпретатор графа. Последовательно выполняет узлы, обрабатывает переходы и условные ветки.

RegisterState / TestContext — хранилище значений регистров и состояния теста. Step’ы обращаются к нему для получения актуальных данных.

Модели Modbus-устройств

SlaveModelBase — базовый класс устройства/слейва. Содержит:

SlaveId — идентификатор слейва Modbus

RegisterItems — список регистров устройства

PollAsync() — метод опроса устройства, реализуется в наследниках

Наследники (например, El60Model) описывают конкретные устройства:

свойства регистров (Current, Voltage и т.д.)

методы быстрого управления (EnableLoad, SetFanPwm, ClearStatistics и т.д.)

заполнение RegisterItems с именами, адресами и категориями регистров

SlaveManager

Менеджер слейвов Modbus. Задачи:

Хранит коллекцию всех слейвов Slaves

Скандирует устройства (ScanAsync) и создаёт модели для каждого типа устройства

Позволяет шагам и мониторам обращаться к конкретным устройствам через их модели

Мониторинг регистров
RegisterMonitor

Циклически опрашивает слейвы через их PollAsync()

Подписывает колбэки на изменения регистров (SubscribeRegister)

Обновляет RegisterState для всех Step’ов

Настраивается через PollInterval

RegisterState

Хранилище актуальных значений регистров

Поддерживает методы:

TryGet(string name, out ushort value) — получение значения по имени регистра

Обновление значений из мониторинга

Разделение Step и Node

Step:

Содержит бизнес-логику (например, сравнение, запись регистра)

Переиспользуемый

Не знает о структуре теста и графе

Node:

Является частью графа выполнения

Управляет переходами (следующий шаг, условная ветка)

Ссылается на Step для выполнения логики

Это разделение позволяет:

Переиспользовать шаги в разных тестах

Визуально строить граф в Nodify

Изменять логику шагов без влияния на структуру графа

Unit-тесты и тестирование

Для тестирования Domain без железа используется эмуляция Modbus:

FakeModbusService — реализует интерфейс IModbusService

Хранит словарь регистров (slaveId, address) => value

Реализует ReadRegistersAsync и WriteRegisterAsync

Unit-тесты проверяют:

правильное обновление RegisterState монитором

работу шагов (CompareStep, RegisterWriteStep)

корректность заполнения и чтения регистров модели (El60Model)

Nodify и UI

UI используется только как редактор графа тестов

Узлы графа отображают шаги и связи между ними

Domain не зависит от Nodify

После завершения редактирования графа, TestExecutor выполняет сценарий полностью без UI

Принципы работы

Инициализация:

Создаётся SlaveManager, подключается к Modbus (или FakeModbusService)

Заполняются модели устройств

Мониторинг:

RegisterMonitor подписывается на регистры и запускает циклический опрос

RegisterState обновляется

Выполнение теста:

TestExecutor выполняет узлы графа

Каждый узел вызывает свой Step

Step обращается к RegisterState для чтения/записи значений

Результаты:

Step возвращает результат (True, False, Next)

Executor управляет переходами между узлами графа
