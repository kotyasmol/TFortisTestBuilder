TFortisTestBuilder
==================

Desktop-приложение для визуального проектирования и выполнения автоматизированных тестовых сценариев для производственного тестирования сетевого оборудования.

Проект разработан как инженерный инструмент для автоматизации проверки готовых устройств, работающих через тестовый стенд и Modbus RTU. Основная идея приложения — заменить ручное выполнение отдельных операций на настраиваемый графовый сценарий, который можно создать визуально, сохранить, повторно использовать и выполнить через единый механизм исполнения.


О проекте
---------

TFortisTestBuilder позволяет собирать тестовый сценарий в виде графа из отдельных нод. Каждая нода отвечает за конкретное действие: запись значения в регистр, проверку диапазона, задержку, HTTP-запрос, цикл по Modbus slave-устройствам или завершение сценария.

Такой подход удобен для производственного тестирования, где процедура проверки часто состоит из повторяющихся шагов:

- подключение устройства к тестовому стенду;
- выбор COM-порта;
- обмен с оборудованием по Modbus RTU;
- запись управляющих значений в регистры;
- чтение измеренных параметров;
- сравнение значений с допустимыми диапазонами;
- принятие решения о результате проверки;
- фиксация итогового статуса.

Приложение помогает сократить количество ручных действий, уменьшить вероятность ошибки оператора и сделать процесс тестирования воспроизводимым.


Основные возможности
--------------------

- визуальное создание тестовых сценариев в виде графа;
- добавление и настройка тестовых нод;
- соединение нод через переходы Next / True / False;
- выполнение сценария через графовый исполнитель;
- подключение к тестовому стенду по COM-порту;
- обмен с устройствами по Modbus RTU;
- сканирование доступных slave-устройств;
- мониторинг регистров;
- сохранение и загрузка тестовых профилей в JSON;
- подсветка активной ноды во время выполнения сценария;
- возможность тестировать доменную логику без реального оборудования.


Поддерживаемые типы нод
-----------------------

Start
    Начало тестового сценария.

End
    Завершение тестового сценария.

Write Register
    Запись значения в Modbus-регистр.

Check Register Range
    Чтение значения регистра и проверка попадания в заданный диапазон.

Delay
    Пауза между операциями.

HTTP Request
    Выполнение HTTP-запроса во внешнюю систему.

Label
    Служебная информационная нода.

For Slaves
    Цикл по диапазону Modbus slave-устройств.

Body Start / Body End
    Границы вложенного графа внутри составной ноды.

Ключевой архитектурный принцип
------------------------------

В проекте разделены понятия Node и Step.

Node отвечает за структуру графа:
- идентификатор ноды;
- связи с другими нодами;
- переходы Next / True / False;
- позицию на рабочем поле;
- участие в визуальном сценарии.

Step отвечает за бизнес-логику:
- выполнить конкретное действие;
- записать регистр;
- считать значение;
- проверить диапазон;
- выполнить задержку;
- вернуть результат выполнения.

Такое разделение позволяет не смешивать UI, графовую структуру и реальное выполнение тестовых операций.


Как работает выполнение графа
-----------------------------

1. Пользователь собирает сценарий в визуальном редакторе.
2. GraphCompiler преобразует визуальные ноды в исполняемый граф.
3. TestExecutor запускает выполнение со стартовой ноды.
4. Каждая нода выполняет связанный с ней ITestStep.
5. Step возвращает результат выполнения.
6. По результату выбирается следующий переход: Next, True или False.
7. UI получает события выполнения и подсвечивает активную ноду.
8. После завершения сценария формируется итоговый статус.

Работа с Modbus RTU
-------------------

Приложение поддерживает обмен с тестовым стендом через Modbus RTU.

Через Modbus выполняются операции:

- поиск доступных slave-устройств;
- чтение holding-регистров;
- запись значений в регистры;
- обновление состояния устройств;
- мониторинг параметров;
- выполнение шагов тестового сценария, связанных с регистрами.

Работа с Modbus вынесена в отдельный сервисный слой, чтобы бизнес-логика тестирования не зависела напрямую от реализации обмена с оборудованием.


Работа без оборудования
-----------------------

Часть логики проекта можно разрабатывать и тестировать без физического стенда.

Это возможно благодаря тому, что:

- доменная логика отделена от UI;
- выполнение сценария работает через абстракции;
- Modbus-взаимодействие вынесено в отдельный сервис;
- шаги тестирования можно проверять отдельно;
- unit-тесты могут использовать fake-реализации вместо реального COM-порта.

Такой подход позволяет разрабатывать и проверять основную логику приложения даже без подключения к тестовому стенду.


Тестирование
------------

В репозитории есть отдельный проект TestBuilder.Tests.

Тестами можно покрывать:

- выполнение отдельных шагов;
- запись значений;
- проверку диапазонов;
- работу с состоянием регистров;
- обновление данных мониторинга;
- выполнение графа;
- поведение моделей устройств.

Запуск тестов:

dotnet test


Запуск проекта
--------------

Требования:

- .NET 8 SDK;
- Windows, Linux или macOS;
- для работы с реальным стендом — COM-порт и устройство с Modbus RTU.

Клонирование репозитория:

git clone https://github.com/kotyasmol/TFortisTestBuilder.git
cd TFortisTestBuilder

Восстановление зависимостей:

dotnet restore

Сборка:

dotnet build

Запуск приложения:

dotnet run --project TestBuilder

Запуск тестов:

dotnet test
